# Devproof AI — project notes

Self-hosted LLM serving + managed-agents platform on Kubernetes. Motivation:
**AI sovereignty** — independence from cloud providers and AI companies. Tagline:
**"Own your scalable AI."** Standing constraint: **everything must scale to
hundreds — maybe thousands — of pods**, and run on the local **docker-desktop**
cluster for dev/test.

Full decision history: `docs/concept/decisions-log.md`. Anthropic-platform
alignment + scale posture: `docs/concept/platform-alignment-and-scale.md`.

## Components
Builds: see BUILD.md (bake-based, version-stamped; scripts/build.sh).
- `operator/` — Go operator (ModelPool, ModelDeployment CRDs) → LLMkube. Go lives at `~/sdk/go/bin` (not on PATH).
- `control-plane/` — Node/TS (Fastify) platform API + session orchestrator, port **:7080**. Postgres-backed (`sql/` = Postgrator-tracked migrations, table `schema_migrations`; baseline `001.do.baseline.sql` is the full schema, consolidated 2026-07-19 from the pre-release 45-file chain and dump-diff-verified). Files are named `NNN.do.<description>.sql`, run EXACTLY ONCE, forward-only (no undo files), and are IMMUTABLE once applied — Postgrator stores an MD5 per file and fails the boot if an applied file changes (`newline: "LF"` + `.gitattributes` keep checksums stable across platforms). New migrations: next number, plain `ALTER TABLE`, no idempotency or backfill guards needed — a migration numbered at or below the DB's current max version is SILENTLY SKIPPED forever (Postgrator only runs version > current, and checksum validation can't flag a never-applied file), so never backfill version numbers; out-of-order merges must renumber to the next free number. The advisory lock in `migrate()` still serializes concurrent migrators. Postgrator seeds a permanent version-0 sentinel row in `schema_migrations` ("nothing applied yet") — don't "clean it up" and don't write a test assuming row 1 is first.
- `console/` — Next.js UI, port **:7090**.
- `catalog/models.yaml` — bundled model catalog (custom models are DB-backed in `catalog_models`).
- `session-runner/` — the session pod: `runner.py` (event-streaming wrapper, platform prompt, checkpoints, Delegate tool) + `devproof_runner/`, the in-process agent loop it drives (pure-Python, built from scratch in-house; was `agent-sdk/devproof_agent_sdk` until the 2026-07-17 merge — plain package now, no pip metadata; the wire/env contract kept its names: `DEVPROOF_SDK_*` envs, `sdkSessionId`, `sessions.sdk_session_id`) — `/v1/messages` SSE client, built-in tools (Bash/Read/Write/Edit/Glob/Grep/WebFetch + Skill), streamable-HTTP MCP client, client-side auto-compaction (chars÷4 estimate vs `DEVPROOF_CONTEXT_WINDOW`, 80% threshold), resume via transcripts in `~/.devproof/sessions`. No subprocess CLI, no Node, no binary identity patching; GBNF-safe tool schemas by construction (the legacy CLI-based runner runtime is gone since dev29). Tests (loop + runner, one suite): `cd session-runner && python -m unittest discover -s tests -p "test_*.py"` (incl. egress/proxy behaviour; `tests/test_runner.py`'s 3 Delegate path tests fail on the Windows host — separator mismatch — and pass inside the image via a bind mount; the image itself ships no tests). The Skill tool is progressive-disclosure (rev 2): `Skill(skill)` = SKILL.md + base dir + bundled-file INDEX (`evals/` excluded); `Skill(skill, path)` lazily loads one bundled file; the console renders Skill calls as their own green "Skill" row ("skill loaded: <name>" / "skill file loaded: <name>/<path>", `rows.ts` kind rewrite). ZERO third-party AI references allowed in `session-runner/` sources (user decision 2026-07-17).

## Running (dev = out-of-cluster; processes die with the session)
In-cluster infra (Postgres, MinIO, gateway, operator, LLMkube subchart) is deployed by the umbrella chart **`helm-charts`** — the single deploy source of truth — into the `devproof` namespace: `helm install devproof helm-charts -n devproof --create-namespace -f helm-charts/values-dev.yaml --skip-schema-validation` (the flag is required for the llmkube subchart schema; re-`helm upgrade` needs `--force-conflicts` because the CP co-owns the `litellm-config` `config.yaml` key at runtime). The dev profile ships NO CP/console pods — those two run out-of-cluster below. No port-forwards needed: `deploy/dev/localhost-lb.yaml` (namespaces = `devproof`, applied once) exposes Postgres `localhost:15432`, MinIO `localhost:19000`, and the gateway `localhost:14000` as LoadBalancer services bound to localhost. Only Prometheus (`19090`) still needs an ad-hoc forward if used.

```

# S3 creds + the gateway/serving namespaces are now explicit (chart deploys everything into `devproof`).
DEVPROOF_RUNNER_IMAGE=devproof/devproofai-session-runner:dev50 \
DEVPROOF_S3_ENDPOINT=http://127.0.0.1:19000 DEVPROOF_S3_BUCKET=devproof-files \
DEVPROOF_S3_ACCESS_KEY=devproof DEVPROOF_S3_SECRET_KEY=devproof-dev-secret \
DEVPROOF_GATEWAY_NAMESPACE=devproof DEVPROOF_SERVING_NAMESPACE=devproof \
npx tsx src/main.ts

# operator - now runs IN-cluster from the chart (ghcr.io/devproof/devproofai-operator:<pinned version>, public GHCR, IfNotPresent; no :latest tags anywhere - user decision 2026-07-19; registry moved Hub->GHCR same day: GHCR has no pull limits).
# `go run ./cmd` is only for iterating on operator code locally; regen CRDs after api/ changes: controller-gen object+crd

# console — ALWAYS a production build (dev mode is too slow)
cd console && npx next build && npx next start -p 7090
```

## Conventions & gotchas
- **Multi-tenancy:** every entity is workspace-scoped via the `X-Devproof-Workspace` header (default `wrkspc_default`), except Serving (pools, deployments, catalog, and external endpoints). Everything under Serving is global; external endpoints dropped workspace_id in migration 022, while pools, deployments, and catalog were always global. Console fetch helpers: `wsGet` (server, `app/lib/api.ts`), `wsHeader`/`apiPost` (client, `app/lib/client.ts`).
- **Workspace lifecycle (spec 2026-07-13):** `workspaces.status` = `active|disabled|deleting|deleted` (migration 029; name uniqueness is a partial index over non-deleted rows). `wrkspc_default` is immutable (no rename/disable/delete). Disabled ⇒ read-only: `workspace-guard.ts` 409s workspace-scoped writes on BOTH API surfaces (exempt: interrupt, runner callbacks, `/v1/workspaces` mgmt; Serving is global and never guarded) and gateway auth JOINs `workspaces.status='active'` (≤30s cache lag); running sessions use the internal key and finish. Delete ⇒ 202 + `workspace-delete.ts` drains batch-wise in FK order (sessions→skills→memory→files→envs→vaults→agents→webhooks→keys-SOFT-deleted→uploads), resumable via boot sweep; the row survives as a TOMBSTONE (id+name keep `gateway_usage` attribution — usage rows are never touched). Progress = live row counts vs the `delete_totals` snapshot (`GET /v1/workspaces/:id/deletion`, polled by the console).
- **IDs** mirror Anthropic: `wrkspc_/apikey_/memstore_/file_/vlt_/env_/skill_/sesn_/agent_`. New ids are `prefix_` + 12-char base36 (`shortId`, `src/id.ts`); legacy 24-hex ids coexist forever (ids are opaque TEXT). Console convention (Managed Agents): the FULL id is the clickable element in lists (names are plain text; environments open their edit modal from the id), and detail pages show the id with a `CopyId` button (`app/lib/copy-id.tsx`).
- **Agent status** (`agents.status`, migration 020): `active|disabled` via `POST /v1/agents/:id/status`; disabled ⇒ 409 `agent disabled` on new sessions AND follow-up messages (running turns finish); console shows a real badge + Disable/Enable toggle and hides disabled agents from create-session dropdowns.

- **Agent delegation (spec 2026-07-17):** `agent_versions.subagents` (migration 041) lists agents a version may push work to; the runner injects a `Delegate` tool via the loop's `AgentOptions.extra_tools` seam (in-process, no egress) whose executor calls `POST/GET /v1/sessions/:id/delegate…` to run a FULL child session (`sessions.parent_session_id`) synchronously. One level only: the CP resolves `subagents` to `[]` for any session with a parent. Interrupt/zombie-fail of the parent interrupts in-flight children. The delegate result's first line is a JSON header `{"session","files"}` (survives result truncation; console parses it for the child link — rows.ts rewrites Delegate rows to the `subagent` kind, like Skill). **Continuation/lock (amendment 2026-07-17b):** children rest at `idle` between turns like any session (no auto-complete substitution) — the parent can keep following up on an idle|failed child by calling Delegate again with `session=<childId>` (`POST /delegate` body gains `session?`, runs the normal follow-up-turn path, contract appended once) until it explicitly locks the child terminal with `complete=true` (`POST /delegate/:childId/complete`, fires `session.completed`); a `completed` child 409s continuation attempts ("locked"); re-completion is idempotent (200, no webhook re-fire).
- **LLM wikis (spec 2026-07-18):** hierarchical, workspace-scoped knowledge corpora (OKF/karpathy pattern — `index.md`/`log.md`, YAML frontmatter, progressive disclosure) that agents mount on the filesystem. Storage mirrors memory stores — `wikis` + `wiki_entries` (migration 044; content in FileStore, object key kind `wiki` = `<ws>/wiki/<wikiId>/<path>`), but wikis attach to **agents** (not sessions) via `agent_versions.wiki_refs` JSONB `[{wikiId, mode:"read"|"write"}]`. Read is the default and unlimited; **write is exclusive** — a wiki has at most ONE writer agent (`validateWikiRefs` in `src/wiki-refs.ts` 409s a second, checked across every agent's LATEST version; the console also hides the `write` mode option for a wiki that already has a writer). The writer agent is **single-session, via a FIFO queue** (`src/writer-queue.ts`, spec 2026-07-18): `gatedLaunch` (`session-actions.ts`) parks any launch (turn 0, follow-up, or delegate-create) whose writer agent already holds the slot into `pending_launches` under model key `wq:<agentId>` (emits `session.waiting {writerAgent}`, session stays queued, invisible to the zombie + model sweeps). `repo.takeNextWriterLaunch` (advisory-locked per agent, releases only when the slot is free) launches the next FIFO — `releaseWriterQueue` RE-RESOLVES the parked payload's wiki mounts first (an earlier writer changed the wiki while this one waited, deleting the fileIds captured at park time → they'd 404 in stage_wikis; live bug sesn_oxmn4wl486d0); triggers = the terminal/interrupt `releaseWriterSlot` hook (snappy) + the reconciler `sweepWriterQueues` (boot + 60s safety net). `writerSlotHeld` = a queued|running session of the agent that is NOT itself writer-parked. So extra writer sessions queue and run one by one — wiki writes never race, and no session is rejected (superseded the old `agentHasActiveSession` 409). Mount: CP resolves refs → `DEVPROOF_WIKIS` (`resolveWikiMounts`, stale refs skipped like missing skills); runner stages at `/mnt/wiki/<name>`, writer syncs back diff-based (read wikis never write back). **Correction flow = existing Delegate/subagents (NO new inbox/tools):** a reader agent lists the writer as a subagent and delegates corrections; the writer's single-session FIFO queue serializes them (a busy writer queues the delegated session, runs one by one). The wiki **structure spec is HARDCODED** in the runner (`WIKI_STRUCTURE`, `wiki_prompt_block`) and always injected into BOTH the writer's and readers' prompts — NOT a user-configurable field (no `structure_guide` column; user decision 2026-07-18). Files of `kind='wiki'` are FK-referenced by `wiki_entries` (like `memory`): `deleteSession` excludes them (`kind NOT IN ('memory','wiki')`), the orphan sweep reclaims unreferenced ones, and `wikiInUse` blocks deleting an attached wiki. Console: "LLM wikis" nav item after Memory stores; `/wikis` list (with per-wiki total byte size from `listWikis` `total_bytes`) + `/wikis/[id]` collapsible folder-tree browser (`+`/`−`, `index.md`/`log.md` pinned on top) with markdown rendering (react-markdown, frontmatter header, Rendered/Raw toggle, in-pane link navigation via `Markdown` `onNavigate`); agent form has a Subagents-style per-wiki read/write picker (write hidden once a wiki has a writer via `listWikis` `writer_agent_id`), prefills the maintainer system prompt + wiki-writer subagent note, and warns that a writer runs one session at a time; a "Wiki Writer" badge shows in the agents list + detail, and the detail lists attached wikis + subagents. Python client (`python-client/`): `client.wikis` (create/list/retrieve/update/tree/content/delete + `.pages.add/delete`) mirrors `memory_stores`, and `Agents.create(wiki_refs=[{wikiId,mode}])`.
- **Model identity (don't regress):** models must not claim a third-party AI identity. Since dev29 the runner drives its own in-process loop (`devproof_runner`), which sends ONLY what the runner passes — no hidden system blocks, no CLI to patch (the old `patch_cli_identity.py` binary-patching went away with the legacy CLI runtime). The remaining rule: the platform prompt anchors identity to the deployment name — and must NEVER name any third-party assistant, even negated (a negated mention made qwen0.5b claim that identity). Residual: tiny models still say "yes" to leading questions ("are you X?" → "Yes, I am." for any X); that's model sycophancy, not a context leak. External clients pointing their own Claude Code at the gateway still send that client's injected blocks (their client, not ours).
- **Session lifecycle:** queued → running (first runner event) → idle|completed|failed. Turn pods are Jobs with `activeDeadlineSeconds` = agent's `turn_deadline_sec` (migration 021, default 7200, form field "Turn deadline") and `backoffLimit: 0`. A pod that dies without reporting (deadline, OOM, CP down at result time) is caught by the **zombie reconciler** (`src/reconciler.ts`, boot + 60s sweep): in-flight session + Job gone/finished + no event within the 120s grace ⇒ `session.failed` event + status failed. `failed` is RESUMABLE — startTurn accepts idle|failed and resumes from the last completed turn's checkpoint (mid-turn progress is lost); the console shows the composer on failed sessions. Runner status posts are **turn-attributed** (`DEVPROOF_TURN` env, echoed as integer `turn`): a post from a stale turn — a pod that outlived an interrupt racing a follow-up — is ignored entirely (no status/checkpoint/webhooks; its rejected salvage tarball is reclaimed), so raced-interrupt salvage is intentionally dropped; posts without `turn` (pre-dev23) apply as before, and the runner's checkpoint-404 retry (dev22) stays as defense-in-depth. Session **Last activity** (`sessions.updated_at`, migration 035) moves on status flips only — queued/running/idle/completed/failed, so every turn boundary — and NOT on token ticks (the 027 trigger stays the sole writer of tokens_in/out). Do not confuse it with the `last_activity` alias in `listStuckSessions` (`repo.ts:337`), which is `GREATEST(created_at, max(session_events.created_at))` and feeds the zombie reconciler — same name, different definition.
- **Launch gate** (2026-07-13): a turn whose local model deployment isn't warm parks its exact startSession payload in `pending_launches` (migration 026; `session.waiting` event; stays queued and invisible to the zombie sweep) instead of starting a pod — the gateway only routes Ready deployments, and CRD-Ready still lags "routable" by config sync + gateway rolling reload + warmup (verified live: releasing on Ready alone reproduces the exact `Invalid model name` 400). Sessions release ONLY via the post-warmup `onModelRouted` hook (`main.ts` `warmedModels`; boot runs one gateway sync so the set repopulates after CP restarts — the operator only triggers on transitions), and the warmup itself waits for `awaitGatewayRollout` first (kubestore; live-bug 2026-07-12: one warmup 200 through the Service is not "routable" — during the rolling config reload, stale route-less gateway replicas stay in rotation for minutes and the released session's first request 400'd off one); the reconciler-cadence sweep is the safety net: releases warmed models, keeps Warming/Deploying waiters, fails sessions whose deployment turned Failed or vanished. Failed deployment at create/follow-up ⇒ fail-fast 409 with the session id (user decision 2026-07-12); interrupt un-parks; failed stays resumable.
- **Context window (2026-07-13):** the operator caps served context at `transform.MaxContextTokens` (32768; KV-cache vs 2Gi CPU pods) and reports the served value in MD `status.effectiveContextTokens`. The CP's `modelPhase` carries it into the launch payload (`contextWindow`, parked payloads included) and `buildTurnJob` renders it as `DEVPROOF_CONTEXT_WINDOW` (renamed from CLAUDE_CODE_AUTO_COMPACT_WINDOW at dev29); the runner loop auto-compacts CLIENT-SIDE against it (serialized-prompt-chars÷4 estimate, 80% threshold, summarize-and-reset; per-tool-result cap = window chars) — sessions no longer depend on SSE usage numbers to compact. Without the env the loop never compacts and a long turn dies on a gateway 400 ContextWindowExceededError (live bug: sesn_543c2c0j8nl3, 33863 > 32768). Console deployment detail shows "N tokens · capped from M"; the Connect-tab small-context warning uses the served value. **The window env alone was NOT enough (fix wave K, 2026-07-16):** the CLI's auto-compact trigger reads context size from the assistant message `usage.input_tokens` (`contextTokens = input+cache_creation+cache_read+output` of the last assistant msg), and LiteLLM's `/v1/messages` chat-completions bridge hardcodes `message_start.usage.input_tokens=0` (real counts arrive only in the final `message_delta`, which the loop accumulator then overwrites from). So the CLI read ~0, never compacted, and turns still overflowed at 30k/39k (live: sesn_uz6d0gdcsiin, sesn_86tjxdge2e32). Fix lives in `helm-charts/files/custom_callbacks.py` (rendered into the `litellm-config` ConfigMap) `async_post_call_streaming_iterator_hook` (`_inject_prompt_estimate`): for SANITIZE-class models it rewrites the streamed SSE frames — **the hook receives already-serialized `bytes` SSE frames, NOT dicts** — raising `input_tokens` in BOTH `message_start` and the final `message_delta` to a prompt estimate (serialized prompt ÷ 4). **DON'T REGRESS: the stream hook is still what makes EXTERNAL Claude Code clients (pointed at the gateway with their own CLI) auto-compact; managed sessions compact client-side in the runner loop since dev29 and no longer need it.** Server-side metering is untouched (reads `standard_logging_object.prompt_tokens`, not the SSE), so the 027 trigger stays the sole writer of session token totals. Verified live: message_start `input_tokens` 0→16 at the wire; a large message_start fires the CLI's `compacting` status (mock-driven, bundled CLI 2.1.202).
- **Events** are push-based via Postgres LISTEN/NOTIFY (`NotifyHub`, `db.ts`), not polling. **Memory sync** is diff-based.
- **Event idempotency (migration 042, dev36):** the runner posts events at-least-once (`post()` retries with jittered backoff), so it mints a `uid` per event (`emit()`); `appendEvents` skips any `uid` already stored for the session (checked under the session `FOR UPDATE` lock). uid-less events (user prompts, pre-dev36 runners) insert unconditionally via the partial unique index `uq_session_events_uid ... WHERE uid IS NOT NULL`. Removing the uid re-opens duplicate transcript rows on a response-lost retry.
- **Session token totals (don't regress, 2026-07-13):** the `gateway_usage` trigger (migration 027) is the SOLE writer of `sessions.tokens_in/out` — one increment + NOTIFY per metered API call, so totals tick live mid-turn and survive failed/interrupted turns. `appendEvents` must never re-accumulate event tokens (the runner's `session.result` usage is display-only; re-adding it double-counts). SSE re-sends the totals frame on token movement (not just status flips); the console header chip blinks per update (`tok-tick`, live sessions only). Totals are accumulated, not recomputed — pruning `gateway_usage` keeps session totals but breaks totals==Usage-page agreement. (Per-step usage from the loop is a dead end: `AssistantMessage.usage` was all zeros on the wire, both local and remote — see spec 2026-07-13-realtime-token-usage. **That zero was the `message_start.usage.input_tokens=0` bridge quirk; fix wave K now injects a prompt estimate there (see the Context-window bullet), so the loop/CLI sees non-zero usage.** This does NOT touch metering: `appendEvents` still never re-accumulates event tokens and the runner's `session.result` usage stays display-only, so the 027 `gateway_usage` trigger remains the SOLE writer of `sessions.tokens_in/out` — no double-counting.)
- **Files** scale via MinIO/S3 (`s3FileStore`), keyed hierarchically `<ws>/<type>/<id>[/<path>]` (spec 2026-07-14: `<ws>/files/<file_id>`, `<ws>/sessions/<sesn>/<file_id>` checkpoints, `<ws>/memory/<store>/<path>`, `<ws>/skills/<skill>/<path>`); the key is stamped at insert into `files.object_key` (migration 033) — NEVER derived on read. FileStore is a dumb KV store; ids/sha256 are minted by routes. Every delete path purges rows+objects via the shared-key rule (`deleteFileRecordById` returns the key only when unshared). Safety net: maintenance sweep (`src/maintenance.ts`, cron in app_settings `maintenance.cron` — legacy `storage.gcCron` honored as fallback, console /settings Maintenance panel + `POST /v1/maintenance/run`) runs ordered cleanups billing→tokens→sessions→files→orphans (spec 2026-07-17-maintenance-design.md): retention prunes of `cost_entries`/`gateway_usage`, full deletes of aged idle/failed/completed sessions (active workspaces only, via shared `deleteSessionFully` in `src/session-delete.ts`), retention deletes of detached upload/output files on `files.last_attached_at` (migration 043: default now(), bumped by the session_files attach trigger), then the orphan sweep (1h grace). Restart CP + console, then confirm all pages 200 and exercise the touched flow against the live cluster. Backend tests: `cd control-plane && npm test` (Node test runner) and `npx tsc --noEmit`.

**`npm test` = `scripts/run-tests.mjs`: the suite, THEN `scripts/sweep-workspaces.ts`.** Most test files create throwaway workspaces (`t-<tag>-${Date.now()}`, see `TEST_WS_PATTERN`) and never delete them; the sweep hard-deletes them in FK order, so a run cleans up its own mess. It runs even on a RED suite (a failed run leaks too) but must never set the exit code — the runner exits with the suite's status, or a broken sweep would report red as green. Name throwaway workspaces with a swept prefix, never one a human would pick. Predecessor (2026-07-14) was `test/workspace-hygiene.test.ts`, a sweep INSIDE the suite with a >1h age guard: it could only collect the PREVIOUS run's leftovers, so ~40 workspaces sat in the dev DB between runs. S3 objects behind swept file rows are left to the GC sweep (`src/gc.ts`, 1h grace).

**`--test-concurrency=1` is on purpose — don't remove it to "speed the suite up"** (it lives in `run-tests.mjs` now, not package.json). The 44 test files share ONE dev database, and `node --test` gives each file its own process, so parallel files raced two ways: (1) MOOT since the 2026-07-19 Postgrator baseline swap — 12 files call `migrate(pool)` at import, but with the schema already baselined there's no DDL left to run (Postgrator finds nothing pending in `schema_migrations`), so the old race — DDL taking `AccessExclusiveLock` while other files' queries held row locks ⇒ `40P01` deadlock (the advisory lock in `migrate()` only serializes migrator-vs-migrator, never migrator-vs-test) — no longer fires; (2) `app_settings` is a singleton row (`id='global'`) that several files write, so any test asserting "costs/limits unchanged" fails when a sibling file writes it. Measured: parallel = 3 of 4 runs fail (13s); `--test-concurrency=1` = 4 of 4 green (128s). Migrating once wouldn't help much — one `migrate()` is only 91ms, and it fixes (1) but not (2). The real fix, if 128s ever hurts, is a schema (or database) per test file; that restores parallelism AND makes both races impossible.
