# Nav tweaks, Ollama Cloud provider, shorter API keys — design

Date: 2026-07-17. Status: approved.

Four small, independent changes. No migrations, no runner/image changes, no
new dependencies.

## 1. Nav: API keys under Dashboard

`console/app/nav.tsx` `GROUPS`: move `["API keys", "/api-keys", "key"]` out of
the "Manage" group into the untitled top group, directly after Dashboard.
Manage keeps: Usage - API, Usage - Sessions, Settings, Workspaces.

## 2. Collapsible "// Manage" group

Only the Manage group is collapsible (a `collapsible: true` flag on the group
entry in `GROUPS`; other groups render exactly as today).

Behavior:
- Collapsed by default.
- Clicking the `// Manage` title toggles it. The title gets a pointer cursor
  and a small chevron that rotates with the state (the `//` CSS prefix stays).
- Auto-expands whenever the current route is inside the group (same
  `isActive` logic over the group's items via `usePathname`), so the active
  item is never hidden. Concretely: the group renders expanded when
  `open || items.some(isActive)`.
- State is plain `useState` in the client `Nav` component — survives
  client-side navigation (Nav stays mounted in the layout), resets to
  collapsed on hard reload. No localStorage.

## 3. Ollama Cloud provider

First-class `ollama` provider for external deployments, mirroring the
OpenRouter preset's shape. Verified live 2026-07-17: `https://ollama.com/v1`
is an OpenAI-compatible surface — `GET /v1/models` returns 200 with an
OpenAI-format list (ids like `gpt-oss:120b`, `qwen3.5:397b`, `kimi-k2.6`);
`POST /v1/chat/completions` returns 401 without a key (Bearer auth). Ollama's
OpenAI-compat layer supports `reasoning_effort` (docs.ollama.com/openai).

- **Console** `console/app/deployments/deploy-modal.tsx` PRESETS:
  `ollama: { label: "Ollama Cloud", base: "https://ollama.com/v1", hint: "gpt-oss:120b" }`.
- **CP** `control-plane/src/server.ts` PROVIDERS:
  `ollama: { base: "https://ollama.com/v1", modelsPath: "/models" }`
  (probe URL verified 200), and add `ollama` to the create-route 400 error
  text listing valid providers.
- **Gateway** `control-plane/src/gateway-config.ts`:
  - `PROVIDER_PREFIX.ollama = "openai"` (OpenAI-compatible wire).
  - Sanitizer stays custom-only (`devproof_sanitize: e.provider === "custom"`
    unchanged) — hosted provider, full-fidelity schemas.
  - **api_base fallback**: `api_base` is only rendered when the row has a
    `base_url` (gateway-config.ts:57). openai/anthropic/openrouter can leave
    it empty because LiteLLM knows their bases, but `ollama` rides the
    `openai/` prefix, so an empty base URL would silently route to
    api.openai.com. Fall back to `https://ollama.com/v1` when
    `e.provider === "ollama"` and `base_url` is null — keeps the existing
    "NULL = provider default" row semantic.
- Key handling, reasoning-effort test connection (flat OpenAI-compatible
  `reasoning_effort` slot), and mandatory context tokens: all standard,
  nothing provider-specific beyond the above.

## 4. Shorter API keys, same entropy

`control-plane/src/repo.ts` `createApiKey` only. Today:
`dpk_` + 48 hex chars = 24 random bytes = 192 bits (52 chars total).

New: `dpk_` + **33 base62 chars** (A–Z a–z 0–9), generated by rejection
sampling over `crypto.randomBytes` (no modulo bias). 62^33 ≈ 2^196.5 ≥ 2^192 —
same-or-better security in 37 chars total.

- Hint format unchanged: `dpk_…` + last 4 chars.
- Gateway auth is a sha256 lookup against `api_keys.key_hash` with no
  format/length check (`custom_callbacks.py` `user_custom_auth`), so existing
  48-hex keys keep working forever. No migration, no gateway change.

## Testing / verification

- `control-plane/test/repo.test.ts`: new-key format — `dpk_` prefix, total
  length 37, charset `[0-9A-Za-z]` after the prefix.
- `control-plane/test/gateway-config.test.ts`: an `ollama` endpoint renders
  `openai/<model_id>` with `devproof_sanitize: false`; without a base_url it
  gets `api_base: https://ollama.com/v1`; with an explicit base_url that wins.
- `cd control-plane && npx tsc --noEmit && npm test`.
- Console production build; restart CP + console; all pages 200.
- Live: nav shows API keys under Dashboard; Manage collapsed by default,
  expands on click and when visiting /settings; create an Ollama Cloud
  endpoint (probe needs a real key — optional); mint a key, confirm 37-char
  format and a successful gateway call with it.

## Out of scope

- Persisting the collapse toggle (localStorage) — rejected, component state
  is enough.
- Ollama native API (`/api/chat`, `ollama/` LiteLLM prefix) — the cloud's
  OpenAI-compat surface is the integration point.
- Re-issuing or migrating existing keys — old format stays valid.
